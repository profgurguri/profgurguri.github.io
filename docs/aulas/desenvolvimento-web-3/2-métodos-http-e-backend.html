Como é feita a comunicação entre o navegador e o computador servidor?

No mundo real e analógico, toda comunicação entre duas partes deve ser regrada, com ambas as partes: emissor e remetente concordando com uma série de premissas antes de ser estabelecida efetiva comunicação entre eles. No mundo digital, isso se reflete. Nós utilizamos protocolos como tais conjuntos de regras.

O que é um protocolo de comunicação?

Um protocolo de comunicação é um conjunto de regras que definem como dispositivos se comunicam e trocam informações.

Iremos utilizar o protocolo HTTP (Hyper Text Transfer Protocol) para permitir a comunicação entre o navegador (ou sua aplicação front-end) com o servidor. Essa comunicação ocorre com a realização de requisições (requests) e suas respectivas respostas (responses) da seguinte forma: (ilustrada aqui como uma conversação)

Cliente (Navegador)
[Resquest] Gostaria de usar o Método "GET" sobre o recurso "/index.html".
Servidor (Sua aplicação Node.js)
[Response] OK (código 200), irei retornar a você o resultado dessa ação.
[Response] Enviando-lhe "<html> ... <title> Minha página principal </title> ..."

Requisições HTTP
...

As requisições HTTP são mensagens enviadas por um cliente (como um navegador web) para um servidor web. Elas iniciam uma comunicação e solicitam a entrega de recursos, como páginas web, imagens ou dados. A estrutura básica de uma requisição HTTP é composta por:

1. Linha de requisição:

  • Método HTTP: Define a ação desejada, como GET para recuperar um recurso, POST para enviar dados, PUT para atualizar um recurso, etc.
  • URL (Uniform Resource Locator): Identifica o recurso específico que está sendo solicitado.
  • Versão do protocolo HTTP: Indica a versão do protocolo utilizada.

2. Cabeçalhos:

  • Chave-valor: Contêm informações adicionais sobre a requisição, como o tipo de conteúdo desejado, cookies de autenticação, idioma do navegador, etc.

3. Corpo da requisição:

  • Opcional: Contém dados adicionais que podem ser enviados ao servidor, como formulários preenchidos ou arquivos.

Após receber a requisição, o servidor processa a solicitação e envia uma resposta de volta ao cliente. A resposta HTTP contém:

  • Código de status: Indica o sucesso ou falha da requisição (200 OK, 404 Not Found, etc.).
  • Cabeçalhos: Contêm informações adicionais sobre a resposta, como o tipo de conteúdo enviado, tamanho do arquivo, etc.
  • Corpo da resposta: Contém o recurso solicitado, como a página web, imagem ou dados.

Alguns dos métodos HTTP mais comuns são:

  • GET: Recupera um recurso do servidor.
  • POST: Envia dados ao servidor para serem processados.
  • PUT: Atualiza um recurso existente no servidor.
  • DELETE: Exclui um recurso do servidor.
  • OPTIONS: Obtêm informações sobre as opções disponíveis para um recurso.
  • HEAD: Obtém os cabeçalhos de resposta para um recurso sem recuperar o corpo da resposta.

Compreender as requisições HTTP é fundamental para desenvolver e entender como a web funciona.

Respostas HTTP
...

As respostas HTTP são mensagens enviadas por um servidor web para um cliente (como um navegador web) em resposta a uma requisição. Elas contêm o resultado da solicitação, que pode ser um recurso específico, como uma página web, imagem ou dados, ou uma mensagem de erro.

A estrutura básica de uma resposta HTTP é composta por:

1. Linha de status:

  • Versão do protocolo HTTP: Indica a versão do protocolo utilizada.
  • Código de status: Indica o sucesso ou falha da requisição (200 OK, 404 Not Found, etc.).
  • Mensagem de status: Uma breve descrição textual do código de status.

2. Cabeçalhos:

  • Chave-valor: Contêm informações adicionais sobre a resposta, como o tipo de conteúdo enviado, tamanho do arquivo, cookies de autenticação, etc.

3. Corpo da resposta:

  • Opcional: Contém o recurso solicitado, como a página web, imagem ou dados, ou uma mensagem de erro detalhada.

Códigos de status HTTP:

Os códigos de status HTTP são divididos em cinco categorias:

  • 200-299: Sucesso
  • 300-399: Redirecionamento
  • 400-499: Erro do cliente
  • 500-599: Erro do servidor

Alguns dos códigos de status mais comuns são:

  • 200 OK: A requisição foi bem-sucedida e o recurso solicitado foi enviado.
  • 404 Not Found: O recurso solicitado não foi encontrado no servidor.
  • 401 Unauthorized: A requisição requer autenticação.
  • 403 Forbidden: O cliente não tem permissão para acessar o recurso solicitado.
  • 500 Internal Server Error: Ocorreu um erro interno no servidor.

Cabeçalhos de resposta HTTP:

Os cabeçalhos de resposta HTTP fornecem informações adicionais sobre a resposta, como:

  • Content-Type: O tipo de conteúdo do corpo da resposta (texto HTML, imagem JPEG, etc.).
Tipos de Conteúdo (Media Types)

O MIME type é o mecanismo para dizer ao cliente a variedade de documentos transmitidos: a extensão de um nome de arquivo não tem significado na web. Acesse "mnd web docs" para ver a documentação deles sobre os MIME Types.

  • Content-Length: O tamanho do corpo da resposta em bytes.
  • Connection: Indica se a conexão entre o cliente e o servidor será mantida aberta para requisições subsequentes.
  • Set-Cookie: Define um cookie no navegador do cliente.

Corpo da resposta:

O corpo da resposta contém o recurso solicitado, como a página web, imagem ou dados, ou uma mensagem de erro detalhada.
Compreender as respostas HTTP é fundamental para desenvolver e entender como a web funciona.

Diferenças entre os métodos GET e POST no navegador:

1. Finalidade:

  • GET: Recupera dados do servidor. É usado para ações como carregar páginas, buscar informações em APIs e enviar formulários simples.
  • POST: Envia dados para o servidor. É usado para ações como criar ou atualizar dados, enviar formulários complexos e realizar transações.

2. Localização dos dados:

  • GET: Os dados são enviados na URL da requisição, visíveis na barra de endereço do navegador.
  • POST: Os dados são enviados no corpo da requisição, ocultos da URL.

3. Segurança:

  • GET: É considerado menos seguro, pois os dados são expostos na URL.
  • POST: É considerado mais seguro, pois os dados são ocultos.

4. Cache:

  • GET: As requisições GET podem ser armazenadas em cache pelo navegador, o que pode melhorar o desempenho.
  • POST: As requisições POST geralmente não são armazenadas em cache.

5. Limite de tamanho:

  • GET: Possui um limite de tamanho para a URL, geralmente 2048 caracteres.
  • POST: Não possui limite de tamanho para os dados enviados.

Em resumo:

  • Use GET quando:
    • Você precisa recuperar dados do servidor.
    • Os dados são simples e não confidenciais.
    • Você deseja que a requisição seja armazenada em cache.
  • Use POST quando:
    • Você precisa enviar dados para o servidor.
    • Os dados são complexos ou confidenciais.
    • Você não precisa que a requisição seja armazenada em cache.

Exemplos:

  • GET: Carregar uma página web, buscar resultados de pesquisa em um site, enviar um formulário de contato simples.
  • POST: Criar uma conta em um site, enviar um formulário de compra, realizar uma transação bancária.

Observação:

  • A escolha entre GET e POST depende da finalidade da requisição e das características dos dados.
  • É importante usar o método correto para garantir a segurança e o bom funcionamento da aplicação.

Respondendo a rotas diferentes
...

Roteamento é o mecanismo que serve ao cliente o conteúdo que ele solicitou. Em aplicações cliente/servidor baseadas na web, o cliente define o conteúdo desejado na URL; especificamente, o path e a querystring.

Normalmente, o roteamento realizado pelo servidor depende do path e da querystring, mas há outras informações disponíveis:

  • cabeçalhos, o domínio, o endereço IP e outras.
    Isso permite aos servidores levar em consideração, por exemplo, a localização física aproximada do usuário ou seu idioma.
